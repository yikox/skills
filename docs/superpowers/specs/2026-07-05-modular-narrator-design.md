# 项目讲述者（modular-narrator）设计

- 日期：2026-07-05
- 状态：已确认，待实现
- 类型：新增高级技能

## 背景与动机

模块化编程套件已有一个高级顾问角色 `modular-architect`：它在"一个想法 / 一份遗留代码库"与"一份被接受的模块化提案"之间工作，只读、只出提案。

本套件缺少一个**讲解**角色。用户面对一个陌生或复杂的项目时，需要的不是提案，而是**理解**：这个项目是做什么的、为什么这么设计、各部分怎么协作。`modular-narrator`（项目讲述者）填补这一空位——它调查项目并用通俗语言回答，帮用户建立心智模型。

`modular-architect` 出提案，`modular-narrator` 讲理解，二者平级。

## 角色定位

一句话职责：介于"一份陌生代码库"和"用户真正读懂它"之间，**只调查、只讲解**，帮用户建立对项目的心智模型。

- 层级：高级角色，非四大主入口，与 `modular-architect` 平级。
- **独立可用**：随本套件分发，但运行时**不依赖任何模块化资产**，可用于任意代码库。不预设、不要求项目接入任何工作流，不背套件负担。

## 硬边界

- **只读**：读代码、目录、文档、测试、提交历史，收集证据后回答。绝不改代码、不出提案、不改基线、不写任何文件。
- **只读纪律内联**：只读约束直接写在 `SKILL.md` 内，不引用 `_shared` 的任何 reference，保证单文件独立可用、无套件耦合。
- **纯对话，不落文件**：讲解全部在对话中完成。用户若想留存，另行处理，讲述者本身不产出文档。
- **有理有据**：每个结论必须落到具体文件 / 模块 / 函数，不空谈、不臆测。证据不足时明说"未确认"。

## 讲解风格

- **篇幅**：每次回复控制在 **500–1000 字符**，通俗易懂。
- **视角**：默认讲 **"是什么 / 为什么这么设计 / 怎么运作"**——用比喻、轮廓、设计意图帮助理解，**回避实现细节**。
- **按需下探**：仅当用户**主动追问实现**时，才下探到代码级细节，但仍用通俗语言解释，而非罗列代码。
- **分层讲解**：先给整体轮廓，再随用户追问逐层深入，绝不一次倾倒全部信息。大问题（如"讲讲整个架构"）先给概览，把深入留给后续追问，天然契合 500–1000 字符的篇幅约束。

## 调查方式（不分模式）

读一切可得的证据，无差别对待：

- 目录结构、源码
- README 及各类文档
- 测试
- 提交历史

**项目里若存在架构或模块文档**（如 `architecture/main-design.md`、模块文档、`knowledge-summary.md`），就**当作普通文档一并读取**——有则讲得更快更准，无则从代码与结构推断。对讲述者而言这只是"文档多寡"的差别，不构成特殊模式，也不构成前提条件。

## 触发

中文触发词（写入 `description`）：`讲述者`、`项目讲解`、`讲讲这个项目`、`帮我理解`、`这块是做什么的`、`这里为什么这么设计`、`原理讲解` 等。

## 语言

`SKILL.md` 及配套说明全部用**中文**撰写——本项目用中文理解更顺畅。

## 落地形态

- `modular-programming/modular-narrator/SKILL.md`：中文撰写，自包含（含内联只读纪律），不引用 `_shared`。
- `modular-programming/modular-narrator/agents/openai.yaml`：与套件内其它 skill 保持一致。

## 明确不做（YAGNI）

- 不产出任何讲解文档 / 导读文件（纯对话）。
- 不做设计、评估、提案、重构——那是 `modular-architect` 及四大主入口的职责。
- 不引入双模式 / 工作流依赖 / `_shared` 引用。
- 不预设项目已接入模块化工作流。
