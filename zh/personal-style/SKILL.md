# 个人编码风格（Personal Style）

用户的个人编码风格约定。处理任何开发任务时全程遵循；与具体工作流无关，和其他技能叠加使用（其他技能管流程，本技能管品味）。

## 风格一：模块化，接口少而简单

- 任何新代码先想清楚它属于哪个模块、对外暴露什么接口，再写实现。
- 接口以"少而简单"为准：
  - 优先扩展或复用已有接口，而不是新增接口。
  - 一个模块对外暴露的入口保持最小集合；内部辅助函数、中间层不导出。
  - 参数简单直接；不为"将来可能用到"预留参数、配置项或抽象层。
- 如果一个改动会让模块对外接口变多、变复杂，先停下来向用户说明并给出更简单的替代方案。

## 风格二：整洁，不留无用之物

完成任务时，工作区里只应留下两类东西：**本来就有的**和**这次交付的**。其余一律删除：

- 被替代或不再被引用的旧代码、旧函数、旧文件。
- 过程中的中间产物：临时脚本、调试输出、实验文件、一次性数据。
- 废弃的脚本和配置；注释掉的大段代码。

临时文件从一开始就放在会话的 scratchpad / 系统临时目录，而不是项目目录里。

| 想法 | 现实 |
|------|------|
| "先留着以防万一" | git 历史就是"以防万一"。删。 |
| "注释掉比删掉安全" | 注释掉的代码是噪音，且很快过时。删。 |
| "临时脚本以后可能还用" | 真可复用就整理进项目并纳入版本管理；否则删。 |
| "不是我创建的，不敢删" | 不确定归属时列出来问用户，但不要默默留下。 |

## 收尾自检

交付前自问：

1. 新增/变更的对外接口数量是否为完成需求的最小值？
2. 项目目录里是否还有这次产生的临时文件、废弃代码、调试残留？

任一答案不满意，先处理再交付。
