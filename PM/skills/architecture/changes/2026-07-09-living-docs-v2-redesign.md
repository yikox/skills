# living-docs v2 重设计：两份文档 + 顺带更新 + git 对账

## Request

用户对 modular-programming 套件反复修改多次仍不满意。共同诊断结论：

- 修改的输入一直是"审视设计时的不对劲感"，而不是真实使用中观察到的失败，导致修改永不收敛；
- 评价对象从"我的项目是否可持续"漂移成了"这套体系本身是否完备优雅"，后者没有客观答案；
- 体系在"太乱→加治理→太重→减治理"之间振荡，因为缺少判定"够了"的锚点。

回归初心，用户重新定义了产品：**这套东西的产品是两份文档，skill 只是维护机制，机制应该薄到几乎看不见。**

1. **项目文档**（PM + 知识沉淀合一）：项目概况、可复用知识、变化记录。此前不满来自样式、排版和缺少文档自身的管理逻辑。
2. **设计文档**（模块地图）：和代码对应，人能看懂，改东西时能快速定位，能用"我要改某某模块的 XX"和 AI 沟通。此前为维护其准确性长出了 L0-L3 门禁等重流程。

## 目标与非目标

目标：

- 两份文档各自有明确的章节结构、写法规则、压缩规则，可读性第一；
- 设计文档与代码有低成本的同步机制（顺带更新 + git 漂移对账），低到不会被绕过；
- 新套件自带验收机制，后续所有修改的输入必须来自验收日志中的证据。

非目标：

- 不做事前流程门禁（L0-L3 分级、模块门、review 流程全部废除）；
- 不做双语镜像（中文单源）；
- 不做默认架构图渲染（图工具随 legacy 冻结）；
- 不兼容旧套件的存储格式（旧项目迁移由 docs-sync 一次性完成，不做双格式共存）。

## 验收判据

新套件"做好了"的客观标准，同时是 docs-acceptance 的检查清单来源：

1. **跨会话恢复**：新会话读完项目文档后能正确说出"现在做到哪、下一步是什么"，用户补充说明的次数趋近于零。
2. **结构不腐化**：连续开发 4-6 周后，模块地图与代码仍对得上（对账时漂移是小修而非重建），用户能在 5 分钟内讲清系统有哪些模块、各管什么。
3. **开销与改动成比例**：日常修改零额外流程；一次对账 10 分钟内。观察信号：出现"绕过流程直接改"即判定机制过重——绕过行为本身就是数据。
4. **产物被消费**：每类产物（概况、当前焦点、知识条目、变更日志、模块文档）在试用期内至少被真实读取并影响过一次决策；从未被消费的产物类型砍掉。
5. **无提醒遵守**：agent 在至少八成会话里不需要提醒就自动遵守"顺带更新"和会话入口。低于此数，减规则而不是加强调。

## 产品设计一：项目文档 `project.md`

单文件，四个固定章节，每章有明确的读者、写法规则、压缩规则：

| 章节 | 读者 | 写法规则 | 压缩规则 |
| --- | --- | --- | --- |
| 概况 | 新会话 / 新人 | 项目是什么、跑起来的命令、关键约定；只在结构性变化时改写 | 不压缩，只改写 |
| 当前焦点 | 下一个会话 | 1-5 行：在做什么、卡在哪、下一步；每次会话结束顺手更新 | 完成即删，移一行进变更日志 |
| 知识 | 未来的自己 | 按主题小节（构建 / 调试 / 环境坑 / 约定）；新知识合并进已有主题，不追加流水账 | 不归档，膨胀时重写合并 |
| 变更日志 | 追溯 | append-only，一条一行：日期 + 一句话 + commit | 超约 50 条或文件超约 15KB 时，旧条目移入 `archives/`，留一行区间摘要 |

可读性硬规则：

- 章节顺序固定，每章开头不超过一屏；
- 禁止在此文件写实现细节叙述（属于 commit message 和设计文档）；
- 归档移动不删证据（日期、commit、结论），只移动或带链接摘要。

## 产品设计二：设计文档 `architecture/`

- `main-design.md`：一段话说系统 + 模块表（每模块 2-3 行：职责、入口）+ 模块协作的文字描述。全文一到两屏。
- `modules/<name>.md`：五节——职责 / 对外接口 / code_paths / 依赖 / 注意点。上限约 80 行；超限说明模块该拆或文档在复述代码。
- 修改规则只有一条：**改到哪个模块，顺手改它的文档；新模块 = 新文件 + main-design 加一行。**
- 废除 baseline/target 双轨、五词关系词表、verified/inferred 标注、模块 kind 分类等治理性结构。code_paths 保留，用途从"事前门禁的定位依据"改为"事后漂移检测的映射表"。

## 机制设计：git 驱动的对账

标记文件 `architecture/.last-sync` 记录上次对齐时的 commit hash（比"设计文档最后一次提交"更准：文档改过一处不代表全部对齐过）。

```text
检查（秒级，会话开始顺手跑，也可手动触发）：
  git diff --name-only <last-sync>..HEAD
  ∩ 各模块 code_paths
  → 列出"代码动了、文档没动"的模块清单

清单非空 → 对账：
  逐模块看 diff，判断文档需不需要更新（确认无需改也算对齐）
  → 完成后 bump .last-sync 到 HEAD
```

配一个小脚本（`_shared/scripts/`）做 diff∩code_paths 的确定性计算，agent 只负责判断与改写。

## 机制设计：三个 skill

新套件目录 `zh/living-docs/`（套件名采用 living-docs；skill 前缀 docs-。评审时可另择，改名是机械操作）：

```text
zh/living-docs/
  _shared/
    assets/       # project.md 模板、module 模板、main-design 模板
    references/   # 文档写法/压缩规则（单文件）、验收清单
    scripts/      # 漂移检测脚本
  docs-init/        # 建两份文档 + 把"顺带更新"规则写入项目 CLAUDE.md/AGENTS.md
  docs-sync/        # 漂移检测 + 对账 + 项目文档归档压缩 + 老项目/旧套件一次性迁移
  docs-acceptance/  # 临时验收员（套件毕业后拆除）
```

- **docs-init**：问最少的问题（项目目标、运行方式），生成两份文档草稿，用户确认后落盘；把顺带更新规则合并进项目 AI 文档（合并不覆盖，首次写入前确认）。
- **docs-sync**：跑漂移检测脚本 → 对账 → 归档压缩检查。也承担旧 modular-programming 项目和无文档老项目的一次性迁移。
- **docs-acceptance**：脚手架。docs-init 和 docs-sync 的 SKILL.md 最后一步硬性写明"完成后必须调用 $docs-acceptance"。它按验收清单逐条自查，给出通过/存疑，问用户确认；用户的否决和备注写入 `acceptance-log.md`。**后续对套件的任何修改，输入必须来自这份日志。** 连续 5 次验收全过、日志不再产生新问题后，拆除此 skill，套件毕业。

从旧套件回收的资产：PM/模块模板骨架、code_paths 定义、"证据不删只移"原则。其余重写，不做兼容。

## 迁移路径

按序执行：

1. **冻结**：打 tag `modular-v1-frozen`；`en/modular-programming/` 与 `zh/modular-programming/` 整体移入 `legacy/`，README 标注"契约冻结，只读参考"；install.sh 不安装 legacy（冻结含义包括不再进入任何人的 skills 目录，避免新旧套件抢触发）。
2. **先写验收**：先落验收清单和 docs-acceptance，再实现 docs-init / docs-sync——验收标准先于被验收物存在，才能约束实现不跑偏。
3. **实现两份文档的模板与规则**，然后是 docs-init、docs-sync。
4. **中文单源**：新套件只有 zh；en 需求出现时从稳定中文版一次性翻译，不做持续镜像。
5. 本仓库自身的 PM/（现为旧格式）在新套件可用后用 docs-sync 迁移，作为第一个真实试用项目。

## 风险

| 风险 | 缓解 |
| --- | --- |
| "顺带更新"被遗忘，漂移累积 | 这正是 git 对账要兜的底；对账频率随漂移量自适应（会话开始必检） |
| 验收 skill 形式化，用户嫌烦每次都过 | 验收清单按 skill 分节、只问关键项；日志证明其价值后自然保留到毕业 |
| 减重过度，复杂项目撑不住 | 由 acceptance-log 的证据决定是否加回能力，而不是预防性设计 |
| 旧套件用户（如有）失去更新 | legacy 冻结但保留在仓库与 tag 中，可随时取用 |

## 验证

- `./install.sh zh --dry-run`：只列出 living-docs 套件 + personal-style，无 legacy 内容；
- 在本仓库跑 docs-init + docs-sync 完整一轮，docs-acceptance 出具首份验收记录；
- 漂移检测脚本单测：构造"代码动、文档没动"的用例，确认清单命中。
